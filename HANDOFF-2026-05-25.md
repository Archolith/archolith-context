## What Happened

A harness comparison-mode benchmark session (`5bb0dc8d`) died after ~10 runs. The proxy entered a runaway error loop during `tool_calls` handling, burning tokens until the Claude session hit its compaction limit (135 compactions, 193 MB JSONL). The user's exact words: *"something is really borken with the proxy right now and I'm not sure how to backtrack to a usable location. this started when we were working on tool_calls."*

## Root Cause (Confirmed)

**`_wrap_response_as_sse()` dropped `tool_calls` deltas.** The function only emitted `role`, `content`, and `finish_reason` deltas when converting non-streaming JSON to SSE. When the model made mixed calls (synthetic + real tools), OpenCode received `finish_reason: "tool_calls"` but no tool call data → infinite retry loop.

**Regression timeline:**
1. `ec29b80` (May 24) — Added synthetic tools. Streaming path had no synthetic detection.
2. `b4214c4` (May 25 12:11) — Forced `stream=false` to upstream when synthetic tools injected, converted result to SSE via `_wrap_response_as_sse()`. **This is where the bug was introduced** — the conversion function never emitted `tool_calls` deltas.
3. `0ec8b92` + `a233c89` (May 25 14:38–15:01) — 6 more fix commits, none fixed the actual format conversion gap.

**Why it worked before synthetic tools:** The `__archolith_recall` tool is handled in the streaming path natively via `stream_with_recall_detection` — it buffers, intercepts, re-sends, and emits proper SSE. No format conversion needed. Synthetic tools ONLY work in the non-streaming path, and the conversion bridge was incomplete.

## Known Risks & Open Questions

1. **`_wrap_response_as_sse` and `_non_streaming_to_sse` still duplicate logic.** If either is updated without the other, the format divergence could re-occur. **Recommend unifying them** — `_non_streaming_to_sse` is the reference; `_wrap_response_as_sse` adds error propagation. The error-propagation wrapper should call `_non_streaming_to_sse` for the body.

2. **Circuit breaker is in-memory only.** Resets on proxy restart. If proxy restarts during an active error loop, the loop resumes until the circuit re-trips. For production multi-worker deployments, consider persisting to LadybugDB.

3. **No integration/E2E testing with live proxy.** The regression fix is verified at the unit level only. A live test with synthetic tools active + streaming client + mixed tool calls would confirm the end-to-end path.

4. **Token budget default of 2M may be aggressive** for long coding sessions that legitimately need more. The user mentioned wanting "a hard top on the proxy to save on tokens if it is just erroring over and over" — the default is a safety valve, not a constraint for normal sessions.

5. **`_fallback_strip_synthetic` preserves non-synthetic tool_calls with `finish_reason: "tool_calls"`.** Now that `_wrap_response_as_sse` correctly emits tool_calls deltas, this path works correctly. But if someone were to remove the tool_calls emission from `_wrap_response_as_sse`, the same regression would reappear.
